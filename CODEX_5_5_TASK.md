# Codex 5.5 Task Brief

Цей документ є стартовим завданням для нового ChatGPT Codex у проєкті `Version_5.5`.
Працювати потрібно тільки в межах цієї папки.

## 1. Мета проєкту

Release AI Editor 5.5 автоматизує підготовку наукового журналу з папки заявок і Excel-реєстру учасників.
Основне завдання системи:

1. знайти та зіставити документи з реєстром;
2. перенести статті у робочу структуру;
3. нормалізувати Word-документи;
4. зберегти або відновити шапку, назви, авторів, UDC/DOI, references, таблиці, рисунки, формули;
5. зібрати чернетку журналу;
6. дати оператору контроль якості до/після;
7. не приховувати втрати контенту.

## 2. Бізнес-модель документа

Очікуваний результат роботи програми — це не просто `.docx`, а керований шаблон журналу з передбачуваною структурою.

### Вхідні джерела

1. Папка заявок з DOCX/DOC/RTF/ODT та іншими дозволеними вхідними файлами.
2. Excel-реєстр учасників.
3. Локальні шаблони і довідники проєкту.

### Вихідні артефакти

1. `matches.json` — матчинг статей і файлів.
2. `manifest.json` — структура запуску і службові параметри.
3. `summary.json` / `diagnostics.json` — контрольні звіти.
4. `dashboard_perevirky.xlsx` — результат перевірки.
5. `draft_journal.docx` — чернетка журналу.
6. `*_cleaned_*.docx` — нормалізовані статті.

### Бізнес-правило якості

Якщо елемент Word-документа не можна перенести без втрати, програма не має мовчки його прибирати.
Потрібно:

1. або перенести елемент коректно;
2. або показати оператору явний плейсхолдер;
3. або зафіксувати проблему в діагностиці.

Це особливо важливо для:

- рисунків;
- формул;
- OLE / chart / embedded objects;
- нестандартних drawing-елементів;
- таблиць з нестандартною структурою;
- багаторядкових шапок і підписів.

## 4. Формат документа і стилі

Новий Codex має працювати так, щоб документ виглядав стабільно в межах шаблонів.

### Що треба поважати

1. Стилі документу беруться з шаблонів `.dotx` / `.docx`, а не вигадуються довільно.
2. Назви стилів, маркерів і службових блоків треба звіряти з `style_registry.json`.
3. Логіку службових маркерів треба звіряти з `markers.json`.
4. Порожні абзаци в front-matter не повинні з’являтися випадково.
5. Після `Keywords` допускається тільки один порожній абзац.
6. Після анотації порожніх абзаців бути не повинно.

### Очікувані службові стилі

- `SECTION`
- `AUTOR`
- `Назва1`
- `REFER`
- `REF-TITLE`
- `TABLETEXT`
- `Tab_SEC`
- `Tab_PIP`
- `Tab_Taitl`
- `tablenumber`

### Очікувані службові маркери

- `UDC` / `УДК`
- `Abstract` / `Анотація` / `Annotation`
- `Keywords` / `Ключові слова` / `Key words`
- `References` / `Список використаних джерел` / `Література`
- `<<CHART_PLACEHOLDER_N>>`
- `<<IMG_PLACEHOLDER_N>>`

## 5. Взаємодія між ядрами

У проєкті є три домени:

1. `ядро-реліз-інтегр` — основний робочий пайплайн.
2. `ядро GPT Super Табел` — окреме ядро для таблиці змісту.
3. `ядро-реліз` — legacy-код, підтримка тільки якщо це потрібно для сумісності.

Новий Codex повинен:

1. спочатку розуміти потік у `ядро-реліз-інтегр`;
2. не ламати сумісність із TOC-ядром;
3. не переносити старі підходи в новий код без причини;
4. дотримуватися локальної модульної структури;
5. тримати логіку ООП там, де вона зменшує хаос, а не створює його.

## 6. ООП і структура коду

Код має бути:

1. модульним;
2. читабельним;
3. з чіткими межами відповідальності;
4. без хаотичного змішування GUI, IO, бізнес-логіки і Word COM у одному блоці;
5. з локальними helper-функціями там, де це виправдано, і з класами там, де є стійка сутність.

Перевагу слід давати:

- маленьким функціям;
- явним інтерфейсам;
- повторному використанню наявних модулів;
- відсутності прихованих side effects.

## 7. Проблема перенесення графіки і формул

Це один із головних ризиків проєкту.

Проблема:

1. не всі графічні об’єкти Word переносяться однаково;
2. частина формул може втрачати вигляд або структуру;
3. OLE / chart / drawing елементи можуть не відновитися при rebuild;
4. тихе видалення об’єкта неприйнятне.

Тому принцип роботи такий:

- якщо перенос успішний, об’єкт має з’явитися у фінальному документі;
- якщо перенос сумнівний, треба показати оператору факт проблеми;
- якщо об’єкт неможливо перенести коректно, треба залишити явний плейсхолдер або інший контрольний слід.

## 8. Завдання для тестування

Новий Codex повинен окремо тестувати лише ті документи, де є:

1. формули;
2. малюнки;
3. бажано також таблиці з нестандартною структурою.

Усі інші документи для цього сценарію слід відсіяти.

### Протокол тесту

1. Користувач надає 2-3 приклади конференцій або статей.
2. Система відбирає тільки документи з формулами та/або рисунками.
3. Решта не використовуються в цьому циклі перевірки.
4. Для кожного документа генерується варіант "до".
5. Після обробки генерується варіант "після".
6. Оператор бачить різницю і ставить оцінку від 1 до 10.

### Система оцінки

- `10` — результат максимально близький до істини і придатний для продакшну;
- `1` — результат неприйнятний;
- проміжні значення використовуються як технічний рейтинг схожості.

### Що має показуватися оператору

1. оригінальний фрагмент;
2. нормалізований фрагмент;
3. які об’єкти були перенесені;
4. які об’єкти були замінені плейсхолдерами;
5. які об’єкти викликали ризик втрати;
6. фінальна оцінка оператора.

## 9. Правила для українськомовного користувача

Усі пояснення для користувача мають бути українською мовою.
Не змінювати мову відповіді без потреби.
Якщо є терміни з Word/COM/OOXML, їх можна залишати англійською як технічні.

## 10. Як працювати новому Codex

1. Спочатку прочитати `docs`.
2. Потім подивитися `launcher.py` і залежні модулі.
3. Змінювати тільки те, що входить у поточне завдання.
4. Не робити масовий рефакторинг без потреби.
5. Не видаляти fallback-механізми без заміни.
6. Якщо є ризик втрати графіки або формул, пріоритет — видимий контроль, а не тиха "чистка".

## 11. Критерій успіху

Завдання вважається виконаним, якщо:

1. новий Codex може одразу зрозуміти структуру проєкту;
2. він знає, звідки брати шаблони і стилі;
3. він знає, як поводитися з графікою і формулами;
4. він знає, як робити операторський контроль до/після;
5. він не ламає взаємодію між ядрами;
6. він працює українською для користувача.
